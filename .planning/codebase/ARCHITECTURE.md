# Architecture

**Analysis Date:** 2026-03-16

## Pattern Overview

**Overall:** MongoDB-centered, event-driven modular SCADA platform with polyglot worker processes.

**Key Characteristics:**
- `realtimeData` in MongoDB is the system hub for telemetry, command metadata, and point state (`docs/schema.md`, `src/cs_data_processor/README.md`).
- Independent protocol/processing services run as separate OS processes or containers and coordinate through database collections (`platform-ubuntu-2404/supervisord.conf`, `demo-docker/docker-compose.yaml`).
- Web access is split between reverse proxy/static delivery and authenticated realtime API service (`platform-ubuntu-2404/json_scada_http.conf`, `src/server_realtime_auth/index.js`).

## Layers

**Edge Acquisition Layer:**
- Purpose: Ingest telemetry and exchange commands with external field protocols and brokers.
- Location: `src/lib60870.netcore/`, `src/dnp3/`, `src/iec61850_client/`, `src/OPC-UA-Client/`, `src/OPC-DA-Client/`, `src/mqtt-sparkplug/`, `src/telegraf-listener/`, `src/plc4x-client/`, `src/S7CommPlusClient/`.
- Contains: C# binaries (`Program.cs`/`.csproj`), Node drivers (`index.js`), and Go clients.
- Depends on: MongoDB config/state (`conf/json-scada.json`, `protocolDriverInstances`, `protocolConnections`).
- Used by: Processing layer via writes into `realtimeData.sourceDataUpdate` and command queue consumption.

**Realtime Processing Layer:**
- Purpose: Normalize incoming updates, apply alarm/quality logic, and perform derived calculations.
- Location: `src/cs_data_processor/`, `src/calculations/`, `src/cs_custom_processor/`.
- Contains: Change stream consumer, cyclic calculation engine, custom processing extension points.
- Depends on: MongoDB change streams and process instance config (`src/cs_data_processor/cs_data_processor.js`, `docs/schema.md`).
- Used by: Historian bridge, server protocol outputs, API/UI reads.

**Service/API Layer:**
- Purpose: Serve realtime data/commands and user-facing APIs with auth.
- Location: `src/server_realtime_auth/`, `src/config_server_for_excel/`, `src/graphql-server/`, `src/mcp-json-scada-db/`.
- Contains: Express API endpoints (`/Invoke/`, `/GetFile`, `/queryJSON`), JWT/RBAC and optional LDAP auth, auxiliary config and MCP services.
- Depends on: MongoDB, PostgreSQL, auth collections (`users`, `roles`).
- Used by: Admin UI, external apps, integration clients.

**Presentation Layer:**
- Purpose: Render operator/admin interfaces and custom web modules.
- Location: `src/AdminUI/`, `src/svgedit/`, `src/svg-display-editor/`, `src/custom-developments/`.
- Contains: Vite-based SPA plus static/legacy viewer pages and SVG tooling assets.
- Depends on: Realtime API and static asset serving via reverse proxy.
- Used by: Operators, engineering/admin users.

**Persistence/Historian Layer:**
- Purpose: Store realtime core state, events, command history, and historian time-series.
- Location: MongoDB collections documented in `docs/schema.md`; SQL relay scripts in `sql/process_pg_hist.sh` and `sql/process_pg_rtdata.sh`.
- Contains: `realtimeData`, `commandsQueue`, `soeData`, `processInstances` and PostgreSQL/TimescaleDB tables.
- Depends on: SQL file output generated by `cs_data_processor`.
- Used by: Grafana, API queries, reporting paths.

## Data Flow

**Telemetry Ingestion (primary flow):**

1. Protocol drivers receive field/broker data and map it to point addressing (`src/mqtt-sparkplug/README.md`, `src/telegraf-listener/README.md`).
2. Drivers update `realtimeData.sourceDataUpdate` in MongoDB (`src/cs_data_processor/README.md`).
3. `cs_data_processor` change stream pipeline detects updates and computes normalized value/quality/alarm fields (`src/cs_data_processor/cs_data_processor.js`).
4. Processor writes realtime updates back to MongoDB and emits historian SQL files (`sql/pg_hist_*.sql`, `sql/pg_rtdata_*.sql`).
5. SQL workers apply those files into PostgreSQL/TimescaleDB (`sql/process_pg_hist.sh`, `sql/process_pg_rtdata.sh`).

**Command Flow:**

1. UI/API posts command intent to `/Invoke/` in `server_realtime_auth` (`src/server_realtime_auth/README.md`).
2. Service inserts command records into `commandsQueue` (`docs/schema.md`).
3. Protocol driver for `protocolSourceConnectionNumber` dispatches to external endpoint and writes ack/result back.
4. Result state is reflected in realtime and UI polling/subscription paths.

**Web Access Flow:**

1. Reverse proxy receives client traffic on HTTP/HTTPS and forwards to realtime server (`platform-ubuntu-2404/json_scada_http.conf`).
2. `server_realtime_auth` serves API endpoints and static resources (or proxy target paths like Grafana).
3. Auth middleware applies JWT/RBAC and optional LDAP binding (`src/server_realtime_auth/index.js`).

**State Management:**
- Shared state is centralized in MongoDB collections; process-level state is ephemeral and re-created on restart.
- Runtime control and redundancy metadata are stored in `processInstances`, `protocolDriverInstances`, and `protocolConnections`.

## Runtime Topology

**Single-host supervisor topology:**
- `supervisord` starts Node, Go, C#, DB relay, and optional drivers as distinct programs (`platform-ubuntu-2404/supervisord.conf`).
- Nginx fronts realtime APIs/UI and can restrict by source IP (`platform-ubuntu-2404/json_scada_http.conf`).

**Containerized demo topology:**
- `demo-docker/docker-compose.yaml` runs MongoDB replica set node, PostgreSQL/TimescaleDB, Grafana, Nginx, and selected drivers/processors.
- Containers mount `conf/`, `log/`, `sql/`, and selected `src/` modules to keep config and generated SQL/log data outside images.

## Key Abstractions

**Point Model (`realtimeData`):**
- Purpose: Canonical representation of tags, source mapping, quality, alarm, and current values.
- Examples: `docs/schema.md` (`realtimeData` document), update handling in `src/cs_data_processor/cs_data_processor.js`.
- Pattern: Source-specific ingestion + centralized normalization.

**Connection/Driver Configuration:**
- Purpose: Decouple protocol runtime behavior from executable code.
- Examples: `protocolDriverInstances` and `protocolConnections` usage in `src/mqtt-sparkplug/README.md`, `src/telegraf-listener/README.md`.
- Pattern: DB-driven runtime config with instance numbering and node affinity.

**Process Instance Control:**
- Purpose: Enable process enablement, redundancy, and node targeting.
- Examples: Mentioned in `src/cs_data_processor/README.md`, `src/cs_custom_processor/README.md`, `src/calculations/README.md`.
- Pattern: Each service self-registers/reads control metadata from MongoDB.

## Entry Points

**Web Realtime Server:**
- Location: `src/server_realtime_auth/index.js`
- Triggers: Supervisor/container startup; HTTP requests from Nginx proxy and clients.
- Responsibilities: Auth, command API, point reads, file retrieval, query endpoints, optional GraphQL initialization.

**Change Stream Processor:**
- Location: `src/cs_data_processor/cs_data_processor.js`
- Triggers: MongoDB change stream updates on `realtimeData.sourceDataUpdate`.
- Responsibilities: Realtime normalization, alarm logic, SOE updates, historian SQL generation.

**Calculation Engine:**
- Location: `src/calculations/calculations.go`
- Triggers: Cyclic scheduler (default 2 s) as process loop.
- Responsibilities: Compute derived points (`origin=calculated`) and write only changed values.

**Protocol Driver Executables:**
- Location: Examples `src/lib60870.netcore/iec104client/Program.cs`, `src/OPC-UA-Client/Program.cs`, `src/mqtt-sparkplug/index.js`.
- Triggers: Supervisor or compose startup and instance enablement.
- Responsibilities: Poll/subscribe external systems, write source updates, dispatch commands.

## Error Handling

**Strategy:** Process-level resilience with logging, retries/looping behavior, and supervisor/container restarts.

**Patterns:**
- Global exception hooks with log output in long-running Node services (`src/server_realtime_auth/index.js`, `src/cs_data_processor/cs_data_processor.js`).
- Continuous worker loops with retry timing for DB/queue processing (`src/cs_data_processor/cs_data_processor.js`).
- Runtime restart policies in supervisor/compose (`platform-ubuntu-2404/supervisord.conf`, `demo-docker/docker-compose.yaml`).

## System Boundaries

**Northbound:**
- Browser/UI consumers through Nginx + realtime API (`src/AdminUI/`, `platform-ubuntu-2404/json_scada_http.conf`).
- Analytics/reporting through PostgreSQL/TimescaleDB and Grafana (`demo-docker/docker-compose.yaml`).

**Southbound:**
- Industrial/device protocols and telemetry brokers through protocol driver modules (`src/lib60870.netcore/`, `src/dnp3/`, `src/OPC-UA-Client/`, `src/mqtt-sparkplug/`, `src/telegraf-listener/`).

**Internal contract boundary:**
- MongoDB collections act as stable contracts between acquisition, processing, API, and historian stages (`docs/schema.md`).

## Cross-Cutting Concerns

**Logging:** File-based logs per process under `log/`, managed by supervisor or container runtime.
**Validation:** Schema expectations are enforced by process logic and docs-driven field contracts in `docs/schema.md`.
**Authentication:** JWT + RBAC with optional LDAP in `src/server_realtime_auth/index.js` and related `src/server_realtime_auth/app/` modules.
